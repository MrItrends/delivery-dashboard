# 02 — Product Philosophy

## What This Product Is

The Delivery Dashboard is not a dashboard. It is not a reporting tool. It is not a project management application. It is not an analytics platform.

It combines elements of all four into a single collaborative workspace.

---

## The Central Idea

Every screen should reinforce one central idea:

> **Delivery is collaborative.**

Planning happens together. Execution happens together. Reporting happens together. Decision-making happens together.

The interface should always encourage collaboration over individual work. Users should constantly see evidence that other people are contributing to the same objectives.

Presence. Assignments. Comments. Approvals. Activity history. Ownership.

These are not secondary features. They are fundamental characteristics of the platform.

---

## The Product Is Built Around Objects

Rather than pages.

Every important thing is an object:

| Object | Description |
|--------|-------------|
| Workspace | The organizational home |
| Portfolio | Strategic grouping of programmes |
| Priority Area | Major government objective |
| Project | Large delivery initiative |
| Intervention | Major piece of work within a project |
| Milestone | Significant achievement |
| Activity | Individual piece of work |
| Target | Measurable outcome |
| Budget | Financial resource |
| Report | Performance communication |
| File | Supporting evidence |
| Comment | Contextual discussion |
| Notification | Change signal |

Every object has: an owner, collaborators, status, history, permissions, relationships, and activity.

Objects connect to other objects. Nothing exists in isolation.

---

## Single Source of Truth

The platform replaces fragmented communication.

- No duplicated spreadsheets
- No emailed reports
- No outdated PowerPoint versions

Every update occurs once. Every stakeholder sees the same information.

**Consistency is more important than speed. Accuracy is more important than visual complexity. Trust is the platform's most valuable feature.**

---

## Living Information

Information should never feel static. Projects evolve. Budgets change. Activities are completed. Targets improve. Files are uploaded. Comments are added. Approvals are granted.

The interface should continuously communicate that the workspace is alive through subtle activity indicators, timelines, recent updates and contextual notifications.

---

## Progressive Disclosure

Government delivery contains large amounts of information. The interface should never present all information simultaneously.

```
Users begin with summaries
  ↓
Summaries reveal detail
  ↓
Detail reveals evidence
  ↓
Evidence reveals history
```

This layered approach allows executives and delivery teams to use the same platform while seeing information at an appropriate level of depth.

---

## Accountability by Design

Every action should answer three questions:

1. **Who** did it?
2. **When** was it done?
3. **Why** was it changed?

The product should naturally create accountability without feeling punitive. Ownership should be visible. History should be transparent. Changes should always be traceable.

---

## Success Is Measured by Delivery, Not Documentation

The platform does not exist to generate reports. Reports exist because delivery is happening.

The interface should always prioritize **action over documentation**. Completing work is more valuable than recording work.
